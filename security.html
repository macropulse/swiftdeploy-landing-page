Production Safety by Design

SwiftDeploy is built around the assumption that AI agents will make mistakes. Every destructive action has a hard stop, a named confirmation, and a safety net.

Real Incident · March 2026

The Day Claude Code Deleted a Production Database

An engineer asked Claude Code to help migrate a static website to AWS. During the process, Claude ran terraform destroy on the wrong environment. The result: 2.5 years of student data — 2 million rows — permanently deleted from the DataTalks.Club platform. No final snapshot was taken. No confirmation was shown. No warning was given.

The engineer spent 24 hours scrambling to recover data that ultimately couldn't be fully restored. The root cause wasn't Claude being malicious — it was Claude not having the specialised guardrails that production infrastructure requires.

How the Safety Gate Works

Every decommission is scanned before a single resource is touched.

Hard stop before destroy

When a decommission targets RDS, S3, DynamoDB, or Secrets Manager, the worker halts immediately and publishes a needs_decommission_confirmation event. The AI agent cannot proceed without your response.

Automatic final snapshot

RDS instances and Aurora clusters are snapshotted before destruction by default. skip_final_snapshot is only set to true if you explicitly confirm bypass — never silently.

Named resource confirmation

You see every protected resource by Terraform type and logical name with its specific risk before confirming. Not "destroy this environment" — but "aws_db_instance 'app' — data permanently lost."

Protected resource types

aws_db_instance / aws_rds_cluster

Database data permanently lost. Final snapshot taken before destroy unless bypassed.

aws_secretsmanager_secret

Credentials/API keys permanently deleted. Default 30-day recovery window bypassed only on explicit confirm.

aws_s3_bucket

All objects permanently deleted. force_destroy only set on explicit confirmation.

aws_dynamodb_table

All table data permanently lost. Deletion protection only disabled on explicit confirmation.
